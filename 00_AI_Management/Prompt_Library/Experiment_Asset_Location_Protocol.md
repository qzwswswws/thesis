# 实验资产定位与证据分层协议

## 1. 适用场景

当任务涉及以下任一内容时，必须先执行本协议：

- `100_remotefiles/MIEXP` 下的离线、伪在线或真实在线实验资产
- 第 4 章、第 5 章与第 6 章中和实验结果、在线验证、系统联动有关的写作
- 需要回答“这个结果在哪里”“哪个版本该写进论文”“某组实验到底算主结果还是探索结果”等问题
- 需要在多个实验版本、多个目录、多个运行资产之间做快速定位与取舍

## 2. 首读顺序

遇到上述任务时，默认按以下顺序建立上下文，不要一开始就盲目扫全目录：

1. `100_remotefiles/MIEXP/00_AI_Management/Output_Drafts/Thesis_Status_MI_Experiment_Online_Integration_Analysis_20260414.md`
2. `100_remotefiles/MIEXP/00_AI_Management/Output_Drafts/MI_Experiment_Asset_Index_20260414.md`
3. `100_remotefiles/MIEXP/00_AI_Management/Output_Drafts/MI_Experiment_Registry_20260414.md`
4. 对应章节正文：
   - `01_Thesis_LaTeX/data/chap04.tex`
   - `01_Thesis_LaTeX/data/chap05.tex`
   - `01_Thesis_LaTeX/data/chap06.tex`
5. 结构冻结稿：
   - `00_AI_Management/Output_Drafts/论文最终结构冻结稿_20260414.md`

只有在这四步之后，才根据需要深入具体实验目录、图表目录或运行时资产目录。

## 3. 证据分层规则

所有实验必须先判断证据层级，再决定写法：

- `A`：主结果证据，可直接支撑正文主线
- `B`：补充证据，可增强稳健性、解释性或外部验证
- `C`：探索性证据，只能写成探索、启发或 pilot
- `D`：失败/止损/过程性材料，不宜写成已完成贡献

禁止把 `C/D` 层实验写成“本文已解决”。

## 4. 在线实验的三层口径

涉及“在线实验”时，必须先区分以下三层，不得混写：

1. **真实在线链路基础验证**
   - 例如 Alpha 可观测性、MARK 标签链路、BLE 短时稳定性。
   - 可支持“在线验证基础已成立”。

2. **真实在线任务级 pilot**
   - 例如双导左右手 MI 的小规模真实系统试跑。
   - 只能支持“已完成初步验证 / pilot”，不能默认支持“已完成强在线结论”。

3. **伪在线 / chronological pseudo-online 适配实验**
   - 本质是公开数据集上的在线评估近似。
   - 必须明确写成 `pseudo-online`、`chronological replay` 或 `在线适配探索`。

## 5. 快速定位输出格式

当回答用户“某实验在哪里”或“哪些结果该写进论文”时，优先用下面这种紧凑格式：

1. 先给一句总判断：主线 / 补充 / 探索 / 不建议硬写
2. 再给一个短表：
   - 实验家族
   - 当前结论
   - 论文位置
   - 快速定位文件
3. 最后才补充风险与边界

避免先给大段背景，再让用户自己去猜文件位置。

## 6. 新实验纳入规则

当发现新增实验、运行记录或远程同步资产时，默认执行以下动作：

1. 判断它属于：
   - 离线主线
   - 伪在线
   - 真实在线链路
   - 真实在线 pilot
   - 失败/止损材料
2. 将其补登记到：
   - `MI_Experiment_Registry_20260414.md`
3. 如果它改变了论文主判断，再更新：
   - `Thesis_Status_MI_Experiment_Online_Integration_Analysis_20260414.md`
4. 如果它只是补充了“放在哪里”的信息，再更新：
   - `MI_Experiment_Asset_Index_20260414.md`

另外，如果新增实验已经影响：

- 第 5 章定位；
- 创新点边界；
- 真实在线与伪在线的区分口径；

则必须同步核对：

- `00_AI_Management/Output_Drafts/论文最终结构冻结稿_20260414.md`

## 7. 论文写作约束

写入论文时必须遵守：

- 第 4 章以离线主线为主体。
- 第 5 章更强调“系统集成、在线联动实现、在线验证基础/初步验证”。
- 第 6 章再承接更强的人机协同、自适应学习、自然交互范式等前瞻性内容。

如果真实在线证据仍不足，优先降标题强度，而不是硬抬结论强度。

## 8. 必要时的实验台账输出

当用户明确提到：

- “有多个版本”
- “有些实验失败了”
- “需要整理哪些该写、哪些不该写”
- “需要高效定位信息”

则默认应输出或更新一个实验台账表，至少包含：

- 实验家族
- 类型（离线 / 伪在线 / 真实在线基础 / 真实在线 pilot / 失败尝试）
- 做了什么
- 当前效果
- 证据级别（A/B/C/D）
- 论文建议位置
- 快速定位文件

不要只给长段分析，而不留下结构化索引。

## 9. 第 5 章必须补写的机制层内容

凡涉及真实在线系统写作，默认还应检查是否已经说明：

1. 预训练权重是否用于冷启动；
2. 后续是否使用本地累计模型继续迭代；
3. 默认预处理链是什么；
4. 每轮训练与下一轮推理的联动方式；
5. 哪些结果文件已经持久化落盘。

如果缺这一层说明，即使有在线 trial 数据，也不能算“系统在线实现”写清楚了。
