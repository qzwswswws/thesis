# 论文章节润色检查清单（2026-04-21）

适用范围：
- 适用于第 3、4、5 章等主体章节的结构调整、语言润色与图表补强。
- 重点服务于“先理清章节逻辑，再统一论文语体，再核对证据支撑”的修改流程。

使用方式：
- 每次开始修改一个章节前，先填写“本章要回答的问题”。
- 修改过程中按“结构检查”“语言检查”“图表与证据检查”逐项勾选。
- 修改完成后，再用“收尾核对”做一轮通读。

---

## 一、本章定位

- [ ] 本章要回答的问题已经用一句完整的话写清楚。
- [ ] 本章的标题与实际内容边界一致，没有外延过大或过小。
- [ ] 本章与前后章节的分工明确，没有与相邻章节大面积重复。
- [ ] 本章开头已经说明本章的任务、对象和展开方式。
- [ ] 本章结尾能够回扣本章问题，并自然引向下一章。

建议先写：

`本章关注的问题是：________________________________________。`

---

## 二、结构检查清单

### 2.1 章节整体结构

- [ ] 各节顺序按论证依赖关系排列，而不是按写作顺序或实验发生顺序排列。
- [ ] 每一节都能回答“这一节存在的必要性是什么”。
- [ ] 各节标题使用同一层次的命名方式，没有一部分偏“方法”、一部分偏“结果”、一部分偏“材料”。
- [ ] 本章没有明显“应该属于别章”的内容滞留。
- [ ] 本章没有重要内容缺少承接，读者不会出现“这一节为什么突然出现”的感觉。

### 2.2 小节内部结构

- [ ] 每个小节开头先交代问题背景或本节目标，而不是直接进入细节。
- [ ] 每个小节都有相对完整的内部结构：问题/目标、实现/方法、证据/结果、结论/边界。
- [ ] 每个小节结尾都有一句总结，说明这一小节得出了什么结论。
- [ ] 段落顺序符合“先总后分、先主后次、先定义后细节”的阅读逻辑。
- [ ] 同一类内容尽量集中，不要在不同小节反复零散出现。

### 2.3 与全文主线的关系

- [ ] 本章与绪论中的研究目标、研究问题、创新点表述保持一致。
- [ ] 文中若需要呼应前文，采用显式复述，不使用含糊指代。
- [ ] 本章所述内容能够为后续章节提供自然铺垫，而不是孤立成段。
- [ ] 本章的实验、系统或方法边界写清楚了，没有超出本章能支撑的结论范围。

---

## 三、语言检查清单

### 3.1 论文语体

- [ ] 已删除明显口语化表达，如“联调”“试跑”“做出来了”“可以看出”“这里说明了”等。
- [ ] 已删除批注性表达，如“不宜”“这里只是”“需要说明的是”“需要指出的是”等高频自我解释句。
- [ ] 已避免使用辩护式句式，如“并非……而是……”“不是……而是……”“不止……更是……”。
- [ ] 已避免过多使用带引号的强调性词语，除非是术语定义或文献原话。
- [ ] 句子以陈述事实和结论为主，而不是写成说明书、项目记录或答辩口吻。

### 3.2 指代与表述精度

- [ ] 没有使用“前一个问题”“这一点”“这一方面”“上述内容”等偷懒式指代替代明确表述。
- [ ] 当需要呼应前文时，直接复述所指对象，而不是只写“第一个问题”“前文所述问题”等。
- [ ] 专业术语前后一致，没有混用近义词造成语义漂移。
- [ ] 硬件术语、算法术语、实验术语区分清楚，例如“通道/导联”“采样/标签”“部署/推理”等。
- [ ] 结果描述尽量定量化，不用“较好”“明显”“稳定”等空泛形容词单独支撑论证。

### 3.3 工程细节转论文表达

- [ ] 程序变量名没有直接进入正文，如 `seq_id` 这类名称已改写为论文语义。
- [ ] 引脚编号、寄存器名、临时调试字段等工程实现细节没有直接写入正文。
- [ ] 工程符号式表达已改成完整文字表达，如“芯片+MCU+蓝牙模块”改为完整系统描述。
- [ ] 截图语言、界面操作语言、程序自动输出语言没有直接搬入论文。
- [ ] 时间、过程记录、开发背景信息仅在论证必要时保留，不用作正文主体。

### 3.4 句式与节奏

- [ ] 长句已拆解，主干清晰，不出现三层以上嵌套修饰。
- [ ] 相邻句子的主语保持稳定，不频繁切换导致阅读断裂。
- [ ] 列举内容时使用平行结构，避免一条写目标、一条写方法、一条写结果。
- [ ] 段首句承担“引导”作用，段尾句承担“收束”作用。
- [ ] 同一段内没有重复说同一层意思。

---

## 四、图表与证据检查清单

### 4.1 图表作用

- [ ] 每张图和每个表都有明确作用，而不是为了“补材料”而存在。
- [ ] 本章至少有一张图或一张表承担“总览/收拢全章逻辑”的功能。
- [ ] 图表标题能够独立说明对象与作用，不依赖正文猜测。
- [ ] 图表与正文之间存在明确引用和解释，不是摆放后不再使用。

### 4.2 图表内容

- [ ] 图中元素命名与正文术语一致。
- [ ] 图中流程、模块和箭头对应实际系统或方法过程，没有视觉上多余但语义上模糊的内容。
- [ ] 表格优先呈现可比较信息，而不是把整段文字挪进表里。
- [ ] 图表中尽量突出“关系”和“逻辑”，而不是堆砌细节。
- [ ] 若图表用于支撑整体性，应能回答“系统如何闭环”“方法如何展开”“实验如何对应”中的至少一个问题。

### 4.3 证据可回溯性

- [ ] 正文中的关键结果都能回溯到本地原始数据、结构化日志、脚本输出或表格汇总。
- [ ] 没有仅依赖截图来支撑关键实验结论。
- [ ] 若正文采用了某一轮实验数据，已确认它是当前应使用的最终版本。
- [ ] 表格中的数字、指标、比值与原始记录一致。
- [ ] 若采用了实验现象描述，已补充相应指标、公式或定量范围。

---

## 五、结果边界检查清单

- [ ] 结论强度与证据强度匹配，没有把“初步验证”写成“充分证明”。
- [ ] 已明确哪些结果是在当前使用条件下成立，哪些尚未做规范化测试。
- [ ] 已避免因表述过强而造成“因果倒置”或“结论越界”。
- [ ] 已避免因表述过弱而削弱本章实际贡献。
- [ ] 对局限性的表达保持学术语气，不写成自我辩护或批注。

建议优先使用：
- “在当前实验条件下……”
- “可作为……的初步依据”
- “结果表明……”
- “由此说明……”
- “本文实现了……，为后续……提供了基础”

尽量少用：
- “并非……而是……”
- “不宜……”
- “不能说明……只能说明……”
- “需要指出的是……”

## 六、章节修改执行流程

### 6.1 开始修改前

- [ ] 先写一句本章要回答的问题。
- [ ] 先判断哪些内容应保留、前移、后移、合并、删除。
- [ ] 先列出本章必须出现的证据、图表和实验。

### 6.2 结构调整阶段

- [ ] 先改标题和小节顺序，再改句子。
- [ ] 先把“总论述”和“总图表”放稳，再处理局部文字。
- [ ] 若相邻章节存在内容重叠，先确定边界，再修改正文。

### 6.3 语言润色阶段

- [ ] 先删口语和批注。
- [ ] 再统一术语与句式。
- [ ] 最后再压缩重复表述和冗余说明。

### 6.4 证据核对阶段

- [ ] 每个实验结论都回到原始数据或汇总表核对一遍。
- [ ] 每张表里的数值、单位、指标定义都检查一遍。
- [ ] 每个图题、表题、图注、表注都重新通读一遍。

### 6.5 收尾阶段

- [ ] 通读本章时，只看标题和每节首尾句，检查主线是否完整。
- [ ] 再通读一遍全文，检查与前后章节的术语和口径是否一致。
- [ ] 最后单独检查是否还残留变量名、引号强调、口语词和偷懒指代。

## 八、可优先推广到后续章节的经验

- 第 4 章优先检查“方法章节是否被写成结果章节”，以及“方法与系统条件是否匹配”。
- 第 5 章优先检查“实验章节是否按验证逻辑组织”，以及“实验结论是否超出证据范围”。
- 若第 2 章继续调整，优先检查“背景综述是否过多侵入方法和系统实现章节”。
- 每一章都应尽量补一张能收拢全章逻辑的图，或一张能收拢主张与证据关系的表。
